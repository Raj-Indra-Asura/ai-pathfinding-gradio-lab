# End-to-End Project Pipeline Guide

**Use this guide with the weekly docs:** each week teaches one component of the same product. The goal is not to memorize isolated files; the goal is to understand how user input becomes a grid, how algorithms search that grid, how results are measured, and how the Gradio app displays everything.

## The Full Product in One Flow

```text
User opens app.py
    ↓
Gradio UI is created in src/pathfinding_lab/ui/gradio_app.py
    ↓
User chooses grid size, obstacles, start, goal, movement mode, algorithm, and heuristic
    ↓
Grid is created by src/pathfinding_lab/core/grid.py
    ↓
Optional obstacles are generated by src/pathfinding_lab/mazes/generators.py
    ↓
Selected algorithm runs from src/pathfinding_lab/algorithms/
    ↓
Algorithm returns SearchResult from src/pathfinding_lab/core/result.py
    ↓
Visualization code turns the grid, path, and visited cells into a figure
    ↓
Metrics and benchmark helpers compare speed, visited nodes, cost, and success
    ↓
Gradio displays the plot, table, and explanation back to the learner
```

Think of every week as adding one box to this flow.

## Core Data Contracts

### Grid

The `Grid` object is the shared map that every other part of the project reads. It owns the board size, obstacle positions, movement rules, and neighbor lookup behavior.

Important responsibilities:

- reject positions outside the grid
- reject blocked obstacle cells
- return valid neighbors for BFS, DFS, Dijkstra, A*, Greedy, and Bidirectional BFS
- provide movement costs for weighted algorithms
- keep coordinate handling consistent across UI, algorithms, tests, and plots

If the grid is wrong, every later layer becomes confusing. That is why Week 2 is the foundation for the whole project.

### SearchResult

Algorithms should not return random tuples with different meanings. They return a consistent result object so the UI, visualization, tests, and benchmarks can all read the same fields.

Typical result information:

- `path`: ordered list from start to goal
- `visited_order`: cells explored while searching
- `path_length`: number of steps in the final route
- `path_cost`: total movement cost
- `nodes_visited`: amount of search work performed
- `runtime_ms`: execution time
- `success`: whether a route was found

This is the handoff point between algorithm logic and product behavior.

### A Note on `Node` and `CellType` (Reference Types)

For speed, **the algorithms operate on lightweight `(row, col)` tuples**, not on `Node` objects.
Positions are stored in sets, dictionaries (`cost_so_far[(row, col)]`), and heaps because tuples
are tiny, hashable, and fast in a tight search loop.

- **`Node`** (`core/node.py`) is an *illustrative / optional* dataclass that bundles a position
  with its `g`, `h`, `f` costs and `parent`. It's useful for *explaining* how A* reasons, but the
  algorithms in this repo track those costs in separate dictionaries instead — so you won't find
  `Node` inside the algorithm loops, and that's intentional.
- **`CellType`** (`core/types.py`) is an enum for *describing and coloring* cells (EMPTY,
  OBSTACLE, START, GOAL, VISITED, PATH), used mainly around visualization rather than inside the
  search.

The mental model the docs teach therefore matches the code: **positions are tuples**; `Node` and
`CellType` are reference helpers. (Week 2 explains this in more depth.)

## How to Trace a Single User Action

When you click a button in the UI, trace it like this:

1. **Input collection**: Gradio reads slider, dropdown, checkbox, and coordinate values.
2. **Validation**: the app checks that start and goal positions are inside the grid and not blocked.
3. **Grid construction**: a `Grid` is created with the chosen dimensions and movement mode.
4. **Obstacle creation**: random or preset obstacles are added while preserving valid start and goal cells.
5. **Algorithm dispatch**: the selected algorithm function is chosen from the algorithms package.
6. **Heuristic dispatch**: A* or Greedy receives the selected heuristic function when needed.
7. **Search execution**: the algorithm explores neighbors, tracks visited nodes, and reconstructs a path.
8. **Result normalization**: the result is represented as a `SearchResult` with common fields.
9. **Visualization**: the grid, path, and visited cells are converted to a color-coded figure.
10. **Metrics display**: runtime, path length, cost, and visited counts are shown in the interface.

This trace is the debugging checklist for the entire application.

## How the Layers Depend on Each Other

### Core before algorithms

Algorithms call grid methods constantly. If `get_neighbors()` includes blocked cells, every search algorithm will look broken even if the algorithm code is correct.

### Algorithms before visualization

Visualization needs two things from an algorithm: the final path and the visited order. If an algorithm only returns the final path, the product can show the answer but cannot teach the search process.

### Visualization before UI confidence

The UI is only useful if the plot explains what happened. A learner should see obstacles, start, goal, explored nodes, and final route without reading internal variables.

### Metrics after correctness

Benchmarking is meaningful only after correctness is stable. Do not compare runtimes until every algorithm returns valid paths and handles no-path cases.

### ML after classical heuristics

A learned heuristic is an experiment layered on top of the classical system. You first need grids, optimal path costs, features, training data, model loading, and A* integration.

## Debugging by Pipeline Stage

| Symptom | Check first | Likely layer |
| --- | --- | --- |
| App will not start | dependencies and package install | Week 1 setup |
| Algorithm walks through walls | obstacle validation and neighbors | Week 2 grid |
| BFS never stops | visited tracking | Week 3 algorithms |
| Dijkstra path cost is strange | movement cost rules | Week 4 weighted search |
| A* visits too many cells | heuristic selection | Week 5-6 heuristics |
| Plot colors look wrong | visualization mapping | Week 7 plotting |
| UI output does not update | callback/state flow | Week 8 Gradio |
| Benchmarks vary too much | random seed and iteration count | Week 9 metrics |
| Learned heuristic fails to load | model path and serialization | Week 10 ML |
| Tests fail after refactor | public API contract changed | Week 11 quality |
| Final demo feels disconnected | missing integration checklist | Week 12 final project |

## Integration Checkpoints

Use these checkpoints to verify that each stage is connected before moving on.

### After Week 2

- You can create a grid from code.
- You can place obstacles.
- You can ask for neighbors of a cell.
- Start and goal cells are valid and unblocked.

### After Week 3

- BFS returns a shortest path on an unweighted grid.
- DFS returns a valid path when one exists.
- Both algorithms return visited cells for visualization.
- No-path scenarios return cleanly instead of crashing.

### After Week 4

- Dijkstra returns path cost, not just path length.
- Weighted movement changes which path is best.
- BFS and Dijkstra match on simple unweighted examples.

### After Week 6

- A* can use Manhattan, Euclidean, Chebyshev, Octile, or weighted heuristics.
- You can explain why the selected heuristic matches the movement mode.
- You can compare visited nodes between Dijkstra and A*.

### After Week 8

- A user can generate a grid, select an algorithm, run it, and see a plot.
- Invalid input shows a useful message.
- The same grid can be reused to compare algorithms.

### After Week 12

- The project can be installed, tested, launched, demonstrated, benchmarked, and explained.
- You can rebuild the product from scratch because you understand each layer and handoff.

## Final Mental Model

The project is a teaching product with four repeated questions:

1. **What map are we searching?** The grid layer answers this.
2. **How do we search it?** The algorithm and heuristic layers answer this.
3. **How do we explain what happened?** The visualization and metrics layers answer this.
4. **How does a learner interact with it?** The Gradio UI answers this.

If you can answer those four questions for every feature, you can extend the project confidently.
